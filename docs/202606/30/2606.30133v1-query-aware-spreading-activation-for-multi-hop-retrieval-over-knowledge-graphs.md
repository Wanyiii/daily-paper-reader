## 摘要
基于知识图谱的检索增强生成（Graph RAG）通过利用图结构，在多跳问答任务上优于平面段落检索。然而，在现有大多数系统中，问题仅用于设定种子节点；后续的遍历变得“查询盲”，仅依赖图结构。例外的是QAFD-RAG，它通过结合边重加权的流扩散求解器实现了查询感知遍历。该架构需要将完整图加载到Python内存中，并使用迭代求解器，迭代次数可变，这增加了与图数据库集成的复杂性。我们提出一种激活扩散方法，通过一个简单的每步语义门实现相同的查询感知遍历：步权重是候选实体描述与问题之间的余弦相似度，且迭代次数固定。整个检索过程——种子映射、传播、Top-K选择和上下文组装——被表示为单个Cypher查询，在一次与Neo4j的往返中执行；图从未离开数据库。在MuSiQue上，我们的方法在精确匹配上（32.80 vs 33.50）与QAFD-RAG相当，并在我们的对比中超越了最强的纯结构性基线HippoRAG，提高了5.3 EM和3.4 F1；在2WikiMultiHopQA上，HippoRAG和QAFD-RAG由于它们的短语节点架构保持了优势。禁用该门的消融实验证实，该门是同时实现F1提升3.6至7.4点和检索延迟降低1.5至4.9倍的原因。

## Abstract
Retrieval-augmented generation built on knowledge graphs (Graph RAG) outperforms flat passage retrieval on multi-hop question answering by leveraging graph structure. In most existing systems, however, the question only sets the seed nodes; the subsequent traversal becomes "query-blind", depending solely on the graph structure. The exception is QAFD-RAG, which implements query-aware traversal via a flow-diffusion solver with combined edge re-weighting. This architecture requires loading the full graph into Python memory and an iterative solver with a variable number of iterations complicating integration with the graph database. We propose a spreading-activation method that achieves the same query-aware traversal with a single per-step semantic gate: the step weight is the cosine similarity between the candidate entity's description and the question, and the number of iterations is fixed. The whole retrieval procedure - seed mapping, propagation, top-K selection and context assembly - is expressed as a single Cypher query executed in one round-trip to Neo4j; the graph never leaves the database. On MuSiQue our method matches QAFD-RAG by exact match (32.80 vs 33.50) and outperforms the strongest purely-structural baseline in our comparison, HippoRAG, by 5.3 EM and 3.4 F1; on 2WikiMultiHopQA HippoRAG and QAFD-RAG retain an advantage due to their phrase-node architectures. An ablation with the gate disabled confirms that the gate is the source of a simultaneous F1 gain of 3.6 to 7.4 points and a retrieval-latency reduction by a factor of 1.5 to 4.9.

## 1. 论文的核心问题与整体含义（研究动机和背景）

-   **核心问题**：基于知识图谱的检索增强生成（Graph RAG）在多跳问答任务中，现有方法（如 GraphRAG、HippoRAG）在检索阶段存在“查询盲点”问题：**问题仅用于选择起始的种子节点，后续的图遍历过程完全依赖图结构，而与原始问题脱钩**。这导致在多跳推理中容易引入不相关节点，降低上下文质量和最终答案的准确性。
-   **现有局限**：唯一实现“查询感知”遍历的 QAFD-RAG 系统，需将全图加载到 Python 内存中，并使用迭代次数不固定的流扩散求解器，这一架构复杂且难以与图数据库（如 Neo4j）高效集成。
-   **本文贡献**：提出一种极其简洁的查询感知传播激活方法。核心思想是：**在每一跳的激活传播中，引入一个“语义门”**，该门的值由候选实体描述与问题的余弦相似度动态控制。整个检索流程被**压缩为一条在 Neo4j 图数据库中一次性执行的 Cypher 查询**，图数据始终驻留数据库，无需离开。这既获得了与 QAFD-RAG 相当的精度，又实现了更快的速度和更简单的部署。

## 2. 论文提出的方法论：核心思想、关键技术细节、算法流程

-   **核心思想**：使用**激活扩散**（Spreading Activation）作为遍历基元，但在每一步传播中注入**查询感知**（Query-Aware）权重。这个权重就是“语义门”（σ(v)），它决定了**哪个邻居节点值得被激活**。
-   **关键技术细节**：
    1.  **图构建**：
        -   **实体与关系抽取**：通过 LLM（Gemini 2.5 Flash）从文档中抽取命名实体（带类型和描述）及二元关系，形成 `:Entity` 和 `:RELATES_TO` 边。
        -   **实体消歧**：通过确定性子集匹配 + LLM 验证的方式合并同实体的不同表面形式。
        -   **存储**：所有实体嵌入（768维）后存入 Neo4j，并建立向量索引。
    2.  **查询感知检索**：
        -   **种子映射**：从问题中提取实体，优先精确匹配，若失败则用向量搜索回退到 `:Entity` 节点。每个种子节点 `v` 获得归一化的初始激活值 `r(0)(v)`（最佳匹配种子为1）。
        -   **传播公式**（核心）：
            在每次迭代 `t` 中，节点 `v` 的激活增量 `Δr(t)(v)` 定义为：
            `Δr(t)(v) = α * σ(v) * Σ_{u in A(t)(v)} r(t-1)(u)`
            其中：
            -   `α`：固定衰减因子（默认0.7）。
            -   `σ(v) = max(cos(e_v, e_q), 0)`：**语义门**，是候选实体 `v` 的描述 embedding `e_v` 与问题 embedding `e_q` 的余弦相似度的非负部分。
            -   `A(t)(v)`：**激活前沿**，是与 `v` 相连且满足两个条件的邻居 `u`：其激活值 `r(t-1)(u) > τ`（阈值0.01），且 `r(t-1)(u) > r(t-1)(v)`（防止循环和低价值分支的单调递减条件）。
        -   **更新策略**：仅在 `Δr(t)(v) > τ` 时，更新 `r(t)(v) = r(t-1)(v) + Δr(t)(v)`，否则保持不变。
        -   **迭代次数**：固定为 `T=3` 次（消融实验表明1-2次提升最大，3次达到最佳，4次饱和甚至下降）。
    3.  **Cypher 实现**：将上述种子映射、3步传播、Top-K 实体和路径选择**全部编码为一条 Cypher 查询**。关键点：
        -   传播状态（resource_map）作为 Cypher MAP 变量通过 `WITH` 子句在查询阶段内传递。
        -   使用 `CALL { ... }` 子查询嵌套和 `apoc.map.merge` 实现迭代。
        -   利用 `gds.similarity.cosine` 计算语义门。
        -   路径选择时，仅保留长度≤4且经过高激活节点的路径。
    4.  **上下文构建**：将选中的实体描述和线性化的关系路径（带方向）拼接后送入 LLM 生成答案。

## 3. 实验设计：数据集、基准与方法对比

-   **数据集**：
    -   **MuSiQue**：多跳（2-4跳）问题组成的基准，构造时确保单跳启发式方法无法正确回答，难度较高。使用了 1,000 道问题的标准样本（消融实验中使用前 500 道）。
    -   **2WikiMultiHopQA**：基于维基百科结构化数据（表格、信息框）构建的多跳基准，包含“桥接”和“比较”两类问题。同样使用 1,000 道样本。
-   **基准方法**：
    -   无图基线：**naive-vector**（纯向量搜索实体描述）。
    -   纯结构性遍历方法：**GraphRAG** (社区检测)，**LightRAG** (一跳邻域聚合)，**HippoRAG** (个性化PageRank)，**PathRAG** (基于流的路径剪枝)。
    -   查询感知方法：**QAFD-RAG** (流扩散+边重加权，最相关对比)。

## 6. 论文的主要结论与发现

1.  **查询感知传播同时提升准确率和速度**：使用语义门 `σ(v)` 代替均匀传播，在 MuSiQue 上 F1 提升 7.4 点（41.68 vs 34.27），在 2Wiki 上提升 3.6 点（55.00 vs 51.44），同时检索延迟降低 1.5 至 4.9 倍。这得益于语义门修剪了低信息分支，缩小了后续迭代的工作集。
2.  **单次 Cypher 查询实现全检索流程**：本文展示了如何将种子映射、固定次数传播、Top-K 选择全部编码为一条 Cypher 查询，无需将图移出数据库，极大简化了部署和数据管理。
3.  **在 MuSiQue 上达到 SOTA 级 EM**：在 MuSiQue 上，精确匹配（32.80）几乎与 QAFD-RAG（33.50）持平，且 F1（41.68）超越了最强纯结构基线 HippoRAG（38.33）。
4.  **在 2Wiki 上结构因素占优**：HippoRAG 和 QAFD-RAG 通过 OpenIE 构建的短语节点架构更适合处理 2Wiki 中的属性类答案（日期、国籍等）和比较型问题，本文方法因实体类型固定而存在局限。

## 7. 优点

-   **方法简洁高效**：用单一余弦相似度门替代了 QAFD-RAG 复杂的边加权和流求解，设计极其巧妙，降低了实现和维护成本。
-   **工程化落地强**：将复杂算法转化为一条 Cypher 查询，是 RAG 系统与图数据库深度结合的典范。这解决了现有系统（需全图入 Python 内存）在生产部署中的重大瓶颈（内存限制、并发安全、备份等）。
-   **结果对比清晰**：通过严格的控制实验（no-query-aware），清晰地向读者展示了核心组件（语义门）带来的净收益，这比仅与外部方法对比更具说服力。
-   **分析透彻**：对 MuSiQue 和 2Wiki 上性能差异的原因（实体粒度、比较问题）给出了深入且合理的结构分析，并指出了未来的改进方向。

## 8. 不足与局限

-   **主要局限：比较条件不匹配**。论文对比的基线方法分数并非在同等实验条件下获得（不同 LLM、可能不同的样本），降低了主表结果的可靠性和客观性。
-   **消融实验样本量有限**：所有消融实验仅在 n=500 问题上进行，论文未详细说明抽样方法，且置信区间（~4%）存在，部分微小的差异可能不显著。
-   **适用场景受限**：在 2WikiMultiHopQA 上性能落后约 15 F1，表明该方法对于属性类答案（如日期、数量）和比较型问题覆盖不足。这源于其固定的实体类型模式和“链式推理”导向的设计中心。
-   **未评估 LLM 鲁棒性**：所有 LLM 调用（抽取、生成）均使用单一模型（Gemini 2.5 Flash），未测试方法对不同 LLM（如 GPT-4o-mini, Claude）的鲁棒性，尤其语义门 `σ(v)` 的效果依赖于实体描述的质量，而描述质量又取决于抽取模型的性能。
