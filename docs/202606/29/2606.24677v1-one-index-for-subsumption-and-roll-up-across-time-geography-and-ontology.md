## 摘要
时间序列、地理空间和本体系统各自维护一个层次结构——日<=月<=年、邮编<=城市、is-a/part-of——并且每个都在独立的存储体中索引。我们观察到这些都是包含偏序集，具有一种重复的工作负载：顺序测试（x是否在y之下？）和层次汇总（对y之下所有对象的度量进行聚合）。我们提出了OEH，一个单一可声明的索引，通过廉价的结构探测，将层次结构编码为嵌套集顺序嵌入（树）或链分解（低宽度DAG），并从单个结构回答包含和索引驻留幺半群汇总。在五个真实层次结构上——基因本体、NCBI分类法（130万）、GeoNames（33万）、一个260万个节点的日历和git提交DAG——OEH在树上的查询延迟与2跳索引相匹配，使用大约一半的空间，构建速度快6-7倍，并增加了2跳无法实现的汇总功能。其汇总功能与TimescaleDB的连续聚合在相同延迟范围内完全匹配，同时也能回答包含问题。在高宽度DAG上，链索引被拒绝，2跳占主导地位。顺序嵌入是经典的；我们的贡献在于统一以及包含和索引驻留汇总上的结构选择索引。

## Abstract
Time-series, geospatial, and ontology systems each maintain a hierarchy -- day <= month <= year, zip <= city, is-a / part-of -- and each indexes it in a separate silo. We observe these are all subsumption posets, with one recurring workload: order testing (is x under y?) and hierarchical roll-up (aggregate a measure over everything under y). We present OEH, a single declarable index that, by a cheap structural probe, encodes a hierarchy as a nested-set order-embedding (trees) or a chain decomposition (low-width DAGs), and answers both subsumption and index-resident monoid roll-up from one structure. On five real hierarchies -- Gene Ontology, NCBI Taxonomy (1.3M), GeoNames (330k), a 2.6M-node calendar, and git commit DAGs -- OEH on trees matches a 2-hop index on query latency using about half the space and building 6--7x faster, and adds roll-up that 2-hop cannot. Its roll-up matches TimescaleDB's continuous aggregates exactly and in the same latency regime, while also answering subsumption. On high-width DAGs the chain index is declined and 2-hop dominates. Order-embedding is classical; our contribution is the unification and the structure-selected index over subsumption and index-resident roll-up.

# 论文总结：一个用于时间、地理和本体上的包含和汇总的索引

## 1. 核心问题与整体含义（研究动机和背景）

时间序列、地理空间和本体系统各自维护层次结构（如日<=月<=年、邮编<=城市、is-a/part-of），但每个系统都在独立的存储体中索引这些层次结构。这些层次结构本质上都是**包含偏序集**，且存在两种重复出现的工作负载：① 顺序测试（x是否在y之下？）；② 层次汇总（对y下所有对象的度量进行聚合）。现有索引方案要么只支持其中一种查询（如传统的2-hop索引只支持包含查询），要么在数据库引擎层面通过连接-分组-聚合来实现汇总，效率低下。本文旨在提出一个**统一的、可声明**的索引，能同时高效支持包含查询和索引驻留的幺半群汇总，跨越时间、地理和本体三个领域。

## 2. 提出的方法论：核心思想、关键技术细节

- **核心思想**：提出 **OEH（Order-Embedded Hierarchy）** 索引，通过一个廉价的结构探针自动选择编码方式，用一个单一结构同时回答包含查询和索引驻留汇总。
- **关键技术细节**：
  - **关系层次结构抽象**：定义带标签的偏序集 (V, ⊑, λ)，查询代数包括包含测试、祖先/后代、LCA 以及层次汇总（折叠幺半群度量）。
  - **结构探针（“旋钮”）**：
    - **树结构 -> 嵌套集（nested-set）顺序嵌入**：使用 DFS 为每个节点赋予区间 [in, out]；包含查询转化为二维包含关系：x ⊑ y ⇔ in(y) ≤ in(x) ∧ out(x) ≤ out(y)。子树为 in 序连续范围，汇总可通过 Fenwick 树实现 O(log n) 范围求和。每个节点只需存储两个整数。
    - **低宽度 DAG -> 链分解（chain decomposition）**：使用路径划分，每个节点记录 (chain, pos)；对于每个节点 v 和链 c，存储 reach[v][c]（可达的最小位置）。包含查询为 O(width)。后代沿链是连续后缀，汇总可通过每个链的后缀和精确实现（不重复计数）。
    - **宽度上限**：链索引空间为 O(n·width)，当宽度超过约 8√n 时，OEH 自动拒绝链模式并回退到 2-hop 索引（PLL）。
  - **索引驻留汇总**：在嵌套集模式下，汇总直接从索引的 Fenwick 范围和中回答（O(log n)）；在链模式下，从每条链的后缀和中回答（O(width)）。这与传统的“索引辅助”方法（引擎执行 join-group-aggregate）不同，后者需要将数据取出后聚合。

## 6. 主要结论与发现

- OEH 索引在**树结构**上，与 2-hop（PLL）查询延迟相当，但空间减半且构建速度快 6–7 倍，并且额外支持汇总功能（2-hop 无法提供）。
- **索引驻留汇总**在大型子树上比引擎级聚合快 3,488 倍（平均 28,851 个后代），且与 TimescaleDB 材料化连续聚合的延迟在同一量级（单微秒级），同时还能回答包含查询。
- **DAG 上链模式很少胜出**：真实的多父节点 DAG（本体、合并历史）通常宽度很高，致使链索引空间过大，自动回退到 2-hop。低宽度多父 DAG 在实际中不常见。
- **制度图谱**：嵌套集适用于树，链适用于低宽度 DAG，高宽度 DAG 应由 2-hop 处理。

## 7. 优点

- **统一抽象**：将时间、地理、本体中的层次结构统一为关系层次结构，并设计一个索引同时支持两种核心查询。
- **结构自适应**：通过简单探针自动选择最优编码方式，无需用户干预。
- **索引驻留汇总**：在索引结构内部直接回答聚合查询，避免了传统 engine join-group-aggregate 的开销，是一个重要的设计创新。
- **实验验证充分**：在多个真实大规模数据集上验证，并与生产级系统（TimescaleDB）及学术基线（PLL、GRAIL）公平对比，证明其实际可行性。

## 8. 不足与局限

- **静态索引**：不支持在线插入/删除操作，动态场景下不如某些增量索引（如 Bulteau 等）。
- **链模式实用范围窄**：真实低宽度多父 DAG 稀少，导致链模式实际很少适用。
- **微小子树汇总性能略差**：当子树极小时 O(log n) 常数开销可能超过线性扫描。
- **TimescaleDB 对比非系统基准**：运行环境是纯 Python 进程，未做系统级优化，只能反映同一量级而不能作精确性能对标。
- **顺序嵌入等基础技术非原创**：论文明确承认 nested-set、dominance 等是经典方法，创新在于统一与索引驻留汇总。
- **部分基线为重新实现**（GRAIL/PLL），但已通过与暴力 oracle 验证精确性；GeoNames 的多父节点（0.9%）被保留了一个规范父节点，可能引入微小近似；链分解采用贪心算法，未保证最小宽度。
